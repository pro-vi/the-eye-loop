Good. This is the right layer.

Taking BALD, AMPLe, drift-diffusion, negative selection, IEC fatigue, and Gemini’s closed generation/comprehension loop as already established, **FRACTING** is the move from a flat posterior to a **nested posterior**. A resolved parent like “dark atmospheric” should not be treated as an answer; it is a **conditioning context** that activates child variables that were previously irrelevant. The closest literature is the overlap of conditional mutual information, tree-structured Bayesian optimization, coarse-to-fine bandits, and local representation learning. ([Proceedings of Machine Learning Research][1])

## 1) Hierarchical BALD: from global uncertainty to conditional predictive information

Flat BALD asks which query maximizes information about the model parameters, and Houlsby et al. explicitly connect that machinery to GP preference learning. For fracting, the natural extension is **conditional** BALD on a node/path: if `π(n)` is the resolved path to node `n`, then the right object is `I(Y; Θ_n | x, D_t, π(n))`, where `Θ_n` are the unresolved local preference variables. Conditional mutual information is exactly the right operator here because `I(X;Y|Z)=0` iff `X` and `Y` are conditionally independent given `Z`, which is the formal version of “is this really a new child axis, or just a correlated echo of the parent?” ([MLG Cambridge][2])

But pure BALD is still slightly too global for Eye Loop. EPIG shows BALD can over-prioritize obscure but downstream-irrelevant inputs, because BALD optimizes information about parameters, not necessarily about the predictions you actually care about. For Eye Loop, the predictions that matter are **future choices inside the currently active subtree**. So the better objective is a **local EPIG**: how much will observing this probe help predict future accept/reject behavior inside this branch? ([Proceedings of Machine Learning Research][3])

This is also a variable-dimensionality problem in disguise. Arc-kernel BO explicitly models which parameters are relevant in a given structure, while tree-structured BO uses leaf-specific Gaussian processes with information sharing along overlapping paths, and a two-stage acquisition that first selects a promising leaf and then optimizes locally within it. Coarse-to-fine contextual bandits make the same move from another angle: explore a coarse low-dimensional space first, and only pay for finer exploration when needed; hierarchical conversational bandits do this explicitly by balancing attribute questions against direct recommendations. That is basically surface-axis vs depth-fract, just in less gothic language. ([Machine Learning Lab][4])

A practical Eye Loop acquisition score is:

[
(n_t^*, x_t^*)=\arg\max_{n \in \mathcal{F}*t,; x \in X_n^{\text{eligible}}}
\left[
P_t(n), I!\big(O_x;\tilde Y*{\mathcal{T}(n)} \mid D_t,\pi(n)\big)
;+;
\beta \sqrt{\frac{\ln(1+t)}{1+N_t(n)}}
;-;
\gamma, d(n)
\right]
]

with `O_x = (Y_x, T_x)` as binary choice plus reaction time, `P_t(n)` the posterior relevance mass of branch `n`, `\tilde Y_{\mathcal{T}(n)}` future responses in subtree `n`, `N_t(n)` the number of probes already spent there, and `d(n)` the depth. Make a probe eligible only if its step size exceeds the node’s local JND, and only allow child nodes onto the frontier once their parent is locked. That is the cleanest formal answer to “surface-level or depth-fract?”: they compete in one frontier, with predictive information, branch relevance, and an exploration bonus in the same score. This is an Eye Loop synthesis built from BALD/EPIG, coarse-to-fine bandits, and tree search. ([Proceedings of Machine Learning Research][3])

If you want the tree-search-native version, HOO is the closest mathematical template. It assigns each region/node an optimistic upper bound

[
U_{h,i}(n)=\hat\mu_{h,i}(n)+\sqrt{\frac{2\ln n}{T_{h,i}(n)}}+\nu_1 \rho^h
]

and then recursively computes `B_{h,i}(n)` from child nodes, always descending toward the highest `B` value. UCT gives the same exploration–exploitation logic on a growing tree. In Eye Loop terms: use a HOO/UCT-style node policy to choose **where** to look next, and conditional EPIG/BALD to choose **what exact probe** to show inside that node. ([arXiv][5])

## 2) Orthogonal decomposition inside a resolved region

A resolved region is not a single scalar. It is a **local manifold**. So orthogonal decomposition should be local, conditional, and supervised.

Recent representation-learning work is useful here because it separates two ideas people love to mush together: **disentanglement** and **orthogonality**. The more relevant target for Eye Loop is orthogonality of factor **subspaces**, not necessarily one-factor-per-coordinate. Recent work argues downstream utility correlates more with orthogonality of factor subspaces than with classic axis-aligned disentanglement; InfoGAN and β-VAE give the core principle of varying one code while keeping the rest fixed, and ISA makes the “independent subspace” view explicit. ([arXiv][6])

First, estimate **residual structure conditioned on the parent path**. Conditional PCA is explicitly designed for context-conditioned representation and is useful when variance given context is smaller than raw variance. If the accepted region is multimodal or curved, mixtures of factor analyzers are better: they jointly do clustering and local dimensionality reduction, letting different local metrics exist in different clusters. That matters because “dark atmospheric” may contain several local styles that should not be crushed into one linear basis. ([ECE Waterloo][7])

Second, separate genuinely independent child factors from correlated leftovers. ICA is specifically meant to reveal hidden factors when PCA or ordinary factor analysis are not enough; HSIC gives a principled independence test; supervised subspace methods like sisPCA use supervision while encouraging disentangled subspaces. In Eye Loop terms, the supervision signal is not class labels from a dataset; it is **fast accepts vs slow local rejects inside the same parent node**. That is exactly where your negative-selection foundation becomes computationally useful instead of merely poetic. ([University of Helsinki][8])

Conjoint-analysis research gives the rule for whether a candidate dimension belongs under a parent or deserves promotion to the surface. An interaction exists when preference for one attribute level depends on the value of another attribute. So if “fog density” only matters once “dark atmospheric” is active, that is a **child axis**. If it matters regardless of parent style, it is a **top-level axis**. ([Carnegie Mellon University][9])

The implementation recipe I would use is: collect accepted samples plus near-boundary rejected samples within the same parent node; encode them in a shared visual/design embedding; residualize locked parent features; run cPCA or local PCA for candidate directions; switch to mixture-of-factor-analyzers if the local cloud is clearly multimodal; then prune/merge candidate directions with HSIC, ICA/ISA, or an orthogonality score like IWO. Only after that do you ask Gemini to name the directions. Let the model label the axes; do **not** let it invent them unchallenged. ([ECE Waterloo][7])

## 3) Prompting Gemini for orthogonal concept generation

I did **not** find strong evidence that generic LLM prompting alone can reliably certify that two concepts are truly orthogonal. What I found is controllable-generation work: prompting can achieve reasonably smooth control and often performs nearly as well as representation engineering, but models still struggle with fine-grained hard constraints, and the intensity descriptors that work for one model do not necessarily transfer to another. So: use Gemini as an **axis proposer** and **image generator**, not as your final theorem prover for independence. ([ACL Anthology][10])

The official Gemini docs support exactly the pieces you need: structured outputs via `response_mime_type: application/json` and `response_json_schema`, image-conditioned generation with image(s)+text to image(s)+text, clear and specific instructions, and descriptive scene prompting rather than keyword soup. Those are the right primitives for a “vary one axis, lock everything else” loop. ([Google AI for Developers][11])

Use Gemini in two stages. Stage one is **axis discovery**: give it the resolved parent path and accepted exemplars, and force JSON output with candidate child axes, low/high anchors, operational control variables, likely correlated drifts, and a hold-constant spec. Stage two is **axis sweep**: attach the best accepted image as reference, lock all resolved parent attributes, and ask for a 5-level sweep on exactly one child axis.

A strong sweep prompt looks like this:

```text
You are performing a one-axis controlled image sweep.

Reference:
Use the attached accepted image as the structural anchor.

Resolved parent path (LOCKED):
- mood: dark atmospheric
- subject identity: same
- composition / framing: same
- camera distance and angle: same
- palette family: same
- lighting direction: same
- motion energy: same
- texture roughness: same
- color temperature: same

Variable child axis (ONLY allowed change):
- fog_density
- levels: [0, 1, 2, 3, 4]
- anchors:
  0 = nearly clear air, background legible
  4 = dense volumetric fog, background mostly obscured

Hard constraints:
- Do not change subject pose, scene layout, temperature, motion blur, grain, palette family, or framing.
- If you cannot satisfy the lock, mark the variant FAIL rather than improvising.

Output:
1) five image variants
2) JSON array with:
   - variant_id
   - axis_level
   - short scene description
   - estimated fog_density (0-4)
   - drift_scores:
       composition
       temperature
       motion
       texture
       palette
   - fail_reason (null if okay)
```

Why this works: it uses image-conditioned generation to preserve structure, structured output to externalize the model’s own audit, descriptive scene language per Gemini’s docs, and an InfoGAN-style “one code varies, the others stay fixed” discipline. If the self-audit shows repeated drift on locked dimensions, the axis is not clean enough; split it, rename it, or demote it from “orthogonal child axis” to “compound correlated style bundle.” ([Google AI for Developers][12])

The practical rule is to operationalize axes as **measurable controls** rather than vibes: fog density, blur magnitude, grain size, color temperature, edge softness, object count, negative-space ratio. “More haunting” is not an axis; it is a custody dispute between several axes. The controllability literature strongly suggests the less ambiguous the control variable, the better your chance of locking everything else. ([ACL Anthology][10])

## 4) Depth vs breadth: the fract decision criterion

Depth vs breadth is just explore/exploit in a better outfit. CoFineUCB explores the coarse low-dimensional feature space first and only expands into the higher-dimensional space as needed; Hier-UCB explicitly models the exploration–exploitation trade-off between asking attribute questions and recommending items; HOO and UCT allocate visits down a tree by optimism. That maps almost literally to Eye Loop: root siblings are surface axes, children are fracts, grandchildren are micro-fracts. ([arXiv][13])

So the formal criterion is not “did the user like this parent branch?” You already have that. The criterion is: **is the expected predictive information of the best child probe, under a locked parent, larger than the best remaining surface probe?** A practical rule is:

[
\text{open children of }n \text{ only if } H_t(n) < h_{\text{lock}}
]

and then go deeper iff

[
\max_{c \in \text{child}(n),, x \in X_c} S_t(c,x)

>

\max_{m \in \text{surface frontier},, x \in X_m} S_t(m,x) + \delta
]

where `S_t` is the frontier score above. That is the clean balance between exploitation (fracting deeper inside a high-probability branch) and exploration (testing a new top-level branch). This is the Eye Loop version of coarse-to-fine UCB. ([arXiv][13])

The stop rule is local too. Stop fracting node `n` when the best child score falls below a node-specific threshold, or when every remaining child probe is below the local JND floor. AMPLe’s halving logic still applies, but only **inside the active subtree**: each child node is a local generalized binary search, and the total session behaves like several small searches, not one giant flat one. Once a node’s remaining uncertainty cell is smaller than what the user can reliably discriminate, more halving is mathematically tidy and behaviorally pointless. ([ACL Anthology][14])

## 5) The JND floor is local, not universal

There is no universal law saying that deeper fract levels always have a finer JND than their parents. The literature is messier and more useful than that. Category learning can improve discrimination along dimensions that predict category membership, and work on visual cortex specifically reports enhancement for boundary-adjacent exemplars. But category learning can also compress within-category differences or leave them unchanged. Meanwhile, subordinate-level training improves fine-grained recognition, and wine experts show better odor recognition without uniformly better raw sensitivity thresholds. In plain English: commitment can sharpen local discrimination, but only along dimensions the task teaches the observer to care about. ([PubMed][15])

So Eye Loop should estimate a **local JND per node and per child axis**. Hold the parent path fixed, generate symmetric small steps along the child axis, fit a psychometric curve from pairwise choices, and use reaction time to mark the near-boundary region. A good operational definition is: the smallest step size that gives stable choice separation and visibly faster RT away from the center. If a candidate child axis never develops a stable local JND, it is not a real child axis yet; it is just the system making expressive hand gestures at the void.

## 6) Representing the fract tree as the Anima

A flat Anima boundary is too small for this job. What you want is a **nested partition** of latent space. Tree-structured BO gives you one natural form: leaf-specific GPs with sharing along common paths. Treed GP models give another: recursively partition the space, fit local GPs in the leaves, and average predictions over tree structure in the posterior. GP preference learning and GP preference elicitation already supply the local probabilistic utility model and value-of-information machinery. Put together, the most natural statistical object for the Eye Loop is a tree where each node holds a local preference model conditioned on its ancestor path. ([Proceedings of Machine Learning Research][1])

Practically, each node should store the path locks, a local preference posterior, candidate child axes/subspaces, node-specific JND estimates, accepted exemplars, near-boundary rejects, and the generation template used to probe that node. Start simple: an explicit decision tree or nested partition with a local GP preference model per node. Upgrade later to a treed GP or GP-gated mixture-of-experts only when the local manifolds become clearly nonstationary or multimodal. Treed GP and GP-gated hierarchical experts are the right escalation path, not the starting line. ([Technical Reports][16])

## 7) Practical dimensionality and session limits

Empirical aesthetics in UI/web contexts usually recovers only a small number of coarse factors: Lavie and Tractinsky’s classical vs expressive split, or the VisAWI four-facet structure of simplicity, diversity, colorfulness, and craftsmanship. That does not mean all visual taste lives in two or four dimensions, but it does mean the **top of the tree should stay small**. A first-session surface with four to six broad axes is much more defensible than dumping twelve on the user and hoping their soul files the paperwork. ([Ben-Gurion University][17])

IEC still gives the practical fatigue bounds. Reviews note that 10–20 individuals and 10–20 generations are common practical settings, maybe up to 30, precisely because user fatigue is the bottleneck. Picbreeder’s escape hatch was branching: it let users start from already-structured ancestors, cutting effort dramatically, and its published-image data show why that matters. Single-user sessions averaged about 20 generations, but cumulative lineages averaged about 151 generations, and 98% of top-rated images required more than 20 cumulative generations. Translation: real depth belongs to **persistent lineages across sessions**, not one heroic swipe binge. ([arXiv][18])

For a **30-swipe** Eye Loop session, my practical recommendation is: expect **depth 2 reliably**, **depth 3 only on one especially strong branch**, and almost never more. A good allocation is 8–10 swipes on breadth over 4–6 top-level regions, 10–12 swipes to lock one winning branch and estimate its child axes, 6–8 swipes on one controlled child-axis sweep, and the final 4–6 on confirmation, local JND estimation, or seeding the next session’s branch. On return sessions, once the top of the tree is already stable, flip that ratio and spend most of the budget on depth. That allocation is an Eye Loop inference from the aesthetics-factor and IEC-fatigue literatures, not a theorem carved into stone tablets. ([Ben-Gurion University][17])

## Implementation-ready answers

**(a) Next-probe formula**

Use a **conditional predictive-information frontier score**:

[
(n_t^*, x_t^*)=\arg\max_{n \in \mathcal{F}*t,; x \in X_n^{\text{eligible}}}
\left[
P_t(n), I!\big((Y_x,T_x);\tilde Y*{\mathcal{T}(n)} \mid D_t,\pi(n)\big)
+
\beta \sqrt{\frac{\ln(1+t)}{1+N_t(n)}}
--------------------------------------

\gamma, d(n)
\right]
]

with eligibility constraints:

1. `Δ_n(x) ≥ \hat JND_n`
2. if `d(n)>0`, the parent must be locked: `H_t(parent(n)) < h_lock`.

That is the clean answer to “surface or depth?” Surface nodes and child nodes sit in the same frontier; whichever has the highest conditional predictive value wins. If you prefer classical BALD over predictive BALD, replace `\tilde Y_{\mathcal T(n)}` with the local node parameters `Θ_n`. ([Proceedings of Machine Learning Research][3])

**(b) Gemini prompt pattern**

Use **two stages**: JSON axis discovery, then image-conditioned one-axis sweep. For the sweep, attach the best accepted image and prompt Gemini to keep the resolved parent path fixed while varying exactly one operational child axis over 5 levels, with structured self-audit JSON for drift. Use descriptive scene language, not keyword salad; use `response_json_schema`; and prefer measurable controls like fog density, blur, grain, or color temperature over vague adjectives. If the audit shows locked-dimension drift, reject the batch and treat the axis as entangled. ([Google AI for Developers][11])

**(c) Practical depth limit for 30 swipes**

In one fresh session: **depth 2 is the safe target**, **depth 3 is possible on one branch if the parent locks early**, and **depth >3 should usually be deferred to later sessions via lineage memory**. The budget should lean breadth-first at the start, then flip toward depth once the top-level branch is stable. That matches the small number of coarse aesthetics factors found empirically and the very stubborn fatigue limits from IEC, plus Picbreeder’s lesson that deep quality usually comes from branching across sessions rather than single-session endurance contests. ([Ben-Gurion University][17])

The shortest way to say the whole thing is this: **Fracting = conditional EPIG on a tree, with local JND floors, local subspace discovery, and one-axis controlled generation.** BALD tells you where disagreement lives; fracting tells you whether that disagreement belongs at the surface or inside the branch you already found.
