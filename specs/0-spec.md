# The Eye Loop -- Hackathon Build Spec

## Zero to Agent: Vercel x DeepMind | March 21, 2026

**Problem Statement:** Statement Three -- AI Applications

**One-line pitch:** A taste amplifier that discovers what you want to build through instinctive selection, not specification.

---

## What It Does

The user types an intent -- "weather app for runners." Instead of generating a spec or writing code, the system generates visual facades and the user swipes left or right. Each swipe -- plus reaction time -- updates a living taste model called the Anima. Agents read the Anima, identify the most uncertain dimension, and generate the next facade to probe it. The process moves from abstract vibes (words, images) to concrete UI (HTML mockups, interactive components). A builder agent assembles surviving artifacts into a coherent prototype continuously in the background. By the time the user finishes swiping, the prototype is already built -- composed entirely from what survived selection.

The user never writes a spec. They discover what they meant by watching what they chose.

> Depth: `research/prior-systems.md` -- Formally: adaptive experimental design over a generative space. PbIG, SwipeGANSpace, GenIR, EvoFIT, and reverse correlation all solve subsets of this same problem. Eye Loop combines all five components (latent space, utility function, query policy, noisy observation, posterior update) with multimodal generation and structured UI. The individual mechanisms are not new; the integration is.

## Core Concepts

**The Eye** -- the user. Selects, never specifies. Binary input (swipe) + continuous latency signal.

> Depth: `research/prior-systems.md`, `research/observation-model.md` -- The Eye is a reverse correlation engine -- a projective instrument. Ambiguous stimuli cause latent preference structure to express through choice. RT is first-class evidence, not UX telemetry; it encodes decision difficulty under drift-diffusion, not preference strength.

**The Anima** -- the accumulated decision boundary. Not a preference list. A tree of resolved and unresolved dimensions. Every agent reads it before generating. Persona (stated intent) and anima (revealed preference) tracked as separate layers -- their divergence is the highest-signal data.

> Depth: `research/fracting.md`, `research/active-preference-learning.md` -- The Anima is a nested partition: each node holds a local preference posterior conditioned on its ancestor path. Tree-structured BO gives the natural form -- leaf-specific models with sharing along common paths. The flat boundary of the original spec is too small; the Anima must be a conditional tree where effective dimensionality changes with the branch.

**Facades** -- visual artifacts generated to trigger instinctive response. Progress through stages: words -> images -> HTML mockups -> interactive snippets. Each carries a stated hypothesis. Each is generated by a specific agent who receives the swipe result.

> Depth: `research/generative-ui.md` -- Dispersion is algorithmic, not decorative. Design Galleries (SIGGRAPH 1997) formalized candidate-set selection: choose parameter vectors whose outputs are perceptually dispersed, then arrange for browsable navigation. Coverage of the output manifold matters more than any single candidate's predicted quality -- the goal is information gain, not approval.

**Fracting** -- when a dimension resolves, the system discovers candidate child axes from local evidence (accepted exemplars + near-boundary rejects). Axes must be empirically tested for independence -- LLM-proposed axes are correlated by default. Gated by: parent lock (H(parent) < threshold), local JND, conditional eligibility. Fracting = conditional search in a variable-dimensional space. Not "deeper on the same axis" -- orthogonal dimensions hidden inside the resolved one, but they must be discovered and validated, not assumed.

**Research basis:** Structurally identical to conditional/hierarchical configuration spaces in Bayesian optimization and AutoML (TPE, arc-kernel BO, tree-structured BO). Once a parent choice resolves, new parameters activate that were previously irrelevant. Conditional MI as axis-validity test: I(Theta_v; y | x, D, c_v) = 0 means the candidate child is a correlated echo of the parent, not a new axis. Local tangent-space estimation (residual PCA on neighborhood samples) discovers candidate directions; HSIC/ICA provides independence testing.

**`[convention]`** Hackathon: LLM proposes candidate axes from accepted exemplar variation (fract detection prompt in specs/1-prompts.md). Axes operationalized as measurable controls (fog density, blur magnitude, color temperature), not vibes. Empirical validation deferred to user feedback -- if preference shifts correlate with drift on locked dimensions, the axis is not clean.

> Depth: `research/fracting.md`, `research/active-preference-learning.md`

**BALD selection** -- two-level controller. Outer: select which node to probe (depth vs breadth) using frontier score -- branch relevance x info gain + exploration bonus - depth penalty. Inner: select which stimulus within that node using conditional BALD or AMPLe halving (pick query whose predicted outcomes are closest to 0.5). Surface nodes and child nodes compete in the same frontier. The frontier score:

    S_t(v,x) = P_t(v) * IG_v(x)
               + beta * sqrt(ln(1+t) / (1+N_t(v)))
               - gamma * d(v)

with eligibility: step size >= local JND; children enter only after parent locks.

**Research basis:** BALD (Houlsby et al.) is the mutual-information acquisition function. Conditional BALD per node conditions on resolved parent path c_v, measuring only information about unresolved local variables. EPIG (Bickford Smith et al.) refines this -- optimizes information about future predictions rather than parameters, avoiding BALD's tendency to chase downstream-irrelevant inputs. HOO/UCT assigns each node an upper-confidence bound and recursively descends toward high-payoff regions. AMPLe provides generalized binary search -- within-node shortcut often faster than full MI computation.

**`[convention]`** Hackathon implementation: distribution flatness (code watches accept/reject distribution across regions; flat = high uncertainty) + builder briefs target the flattest region. Not LLM-estimated uncertainty -- LLM confidence is unreliable for aesthetic judgment.

> Depth: `research/active-preference-learning.md`

## Agent Roster

V0 runs on a role-tiered Anthropic Claude runtime, reached through the Claude Code OAuth header path (`src/lib/server/ai.ts`, env var `CLAUDE_CODE_OAUTH_TOKEN`). Defaults keep the validated Haiku/Sonnet split, but each role can now be tuned independently via env: `SCOUT_MODEL_ID`, `ORACLE_MODEL_ID`, `BUILDER_MODEL_ID`, `REVEAL_MODEL_ID`. The original Gemini three-tier allocation is preserved in *Appendix: historical Gemini three-tier design* in the Tech Stack section for future image/motion work.

**Oracle** -- `ORACLE_MODEL` (default: Haiku 4.5). The brain. Manages agent lifecycle, routes swipe results, assigns probe briefs, monitors queue health, triggers compaction. Never generates facades.

**Builder** -- `BUILDER_MODEL` (default: Haiku 4.5) for incremental draft rebuilds, `REVEAL_MODEL` (default: Sonnet 4.6) for the final reveal synthesis. The architect. Runs continuously from first surviving artifact. Maintains a living draft prototype. Identifies construction ambiguities and writes probe briefs. Has the best view of what the system doesn't yet know -- because it's the one trying to build with incomplete information. Never generates facades. Never swipe-facing.

**Scout (x6 fixed roster: Iris, Prism, Lumen, Aura, Facet, Echo)** -- `SCOUT_MODEL` (default: Haiku 4.5) for both the word-probe generation pass and the optional mockup-HTML rendering pass. The hands. Each runs a Ralph-style loop: generate -> push -> wait -> receive feedback -> decide -> loop. Pulls probe briefs from the builder first, self-assigns from Anima uncertainty if no briefs pending. Fract-style child spawning is explicitly cut from V0 (see `specs/2-v0-spec.md`).

**Compactor** -- function within oracle, reserved for V1. Explicitly cut from V0 per `specs/2-v0-spec.md`; oracle synthesis every 4 swipes (`ORACLE_MODEL`) covers the evidence-rollup role in the Akinator pivot. When the full compactor lands it would run every 5 swipes or on contradiction detection via `ORACLE_MODEL`, merging convergent evidence and promoting contradictions to new probe briefs.

> Depth: `research/iec-fatigue.md` -- Approximately 30-40 binary decisions before fatigue. Oracle is 80% code, 20% LLM. Most oracle functions (queue health, freshness pruning, scout retirement, event routing) are pure code. LLM used only for compaction, fract detection, and stuck detection.

## Architecture

**Status note:** the codebase now ships the per-session reservoir runtime. `EyeLoopSession` owns session state and a session-local event stream; `runtime.ts` owns bootstrap, refill, scout production, oracle synthesis, builder patching, and reveal prep.

### Hot Session Runtime

The user may wait through a brief loading / warmup screen. Once they enter the swipe loop, the next card should already be ready. That requires a session-owned pipeline, not scout-owned one-card blocking loops.

**Runtime states**

`creating -> warming -> ready -> swiping -> reveal-prepared -> revealing -> revealed`

- `creating`: allocate session id, initialize session object, seed empty draft + initial taste snapshot
- `warming`: prefill the shared deck and first draft before the client enters swiping
- `ready`: return bootstrap snapshot (`sessionId`, initial facades, draft, synthesis, stage, queue stats)
- `swiping`: user consumes cards from the ready deck while adaptation runs in the background
- `reveal-prepared`: reveal artifact already exists or is close enough that threshold crossing is just a state change

**Ownership split**

- `SessionRegistry`: in-memory map of `sessionId -> EyeLoopSession`
- `EyeLoopSession`: evidence, swipe count, stage, taste version, draft, reveal state, and worker coordination
- `Reservoir`: card inventory, draw order, queue stats, replacement policy, stale eviction
- `Scouts`: stateless producers that generate candidates against a session snapshot; they do not own exactly one card each
- `Oracle`: coalesced taste updates, synthesis cadence, scout assignments, and freshness version bumps
- `Builder`: async draft/reveal pipeline with its own work queue; never blocks swipe delivery
- `SSE`: session-scoped stream and replay surface, not a single global bus for all sessions

**Hot path vs async path**

- Hot swipe path: record swipe, validate session/card, append compact evidence, increment count, consume current card, reveal next ready card immediately
- Async path: oracle synthesis, scout generation, draft patching, stale-card replacement, and reveal preparation
- Rule: no LLM call belongs between a swipe and the next visible facade

**Reservoir policy**

- Warm to at least `12` ready cards before entering `ready`
- Maintain `16-24` ready cards during swiping
- Treat `8` as the hard low-water mark that triggers aggressive refill
- Expect roughly `55-70` total generated candidates across a 42-swipe session so stale or low-value cards can be discarded without starving the feed

These are operating targets, not sacred constants. The real invariant is that the user should almost never wait for the next card.

**Freshness contract**

Every queued facade should carry:

- `tasteVersion`
- `createdAt`
- `generationReason`

On each oracle update, prefer the newest cards, allow at most `1-2` taste versions of lag for continuity, and demote / replace anything older unless the reservoir is close to empty. Prefetch is useful only if stale work can be evicted cheaply.

**Reveal contract**

- Start reveal preparation early, around swipes `8-12`
- Rebuild or patch reveal material at each synthesis cadence
- Force a dedicated reveal-prep pass by about swipe `36`
- At swipe `42`, transition into reveal using an already-prepared artifact; only final polish may remain on the premium model

The reveal threshold is therefore a presentation gate, not the moment heavy generation begins.

**Landed files**

- `src/lib/server/session/registry.ts`: in-memory session map
- `src/lib/server/session/eye-loop-session.ts`: session state, event replay, evidence serialization
- `src/lib/server/session/runtime.ts`: reservoir refill, scout production, oracle synthesis, builder patching, reveal prep
- `src/routes/api/session/+server.ts`: warm bootstrap response
- `src/routes/api/stream/+server.ts`: session-scoped SSE replay
- `src/routes/api/swipe/+server.ts`: append-only swipe hot path

### Communication Model

Agents never call each other directly. Three communication channels:

**Session state** -- the evidence list, synthesis, reservoir, probe queue, draft prototype, and reveal state live on `EyeLoopSession`. All generation work receives a session snapshot and writes back through session methods.

**Session event stream** -- fire-and-forget notifications. `facade-ready`, `swipe-result`, `evidence-updated`, `synthesis-updated`, `draft-updated`, `queue-updated`, `reveal-prepared`, `stage-changed`, and `error`. `/api/stream?sessionId=...` replays current session state and then forwards live session events.

**Probe queue** -- the builder's voice. Builder identifies construction ambiguities, writes detailed probe briefs, pushes to the session queue. Scouts consume briefs while filling the reservoir; if no brief exists, they self-assign from synthesis and queue gaps.

Tools go outward (runtime -> Anthropic via Vercel AI SDK `generateText`). Events go sideways through the session event stream. Shared state goes through `EyeLoopSession`.

## The Session Class

Per-session server-side state object. The registry is process-local for V0; every HTTP/SSE route resolves the session by `sessionId`.

```
EyeLoopSession
  State:
    .synthesis        current taste synthesis
    .facades          ready reservoir
    .probes           probe brief queue (builder pushes, scouts pull)
    .agents           registry of active agent states
    .draft            builder's living prototype
    .reveal           prepared/final reveal state
    .tasteVersion     freshness version for queued facades

  Derived (getters):
    .nextProbe        shift from probe queue, or null
    .queueStats       ready/target/min/max/lowWater/pending/stale
    .concretenessFloor word until 4 evidence, then mockup
    .sessionMedianLatency RT bucket baseline

  Methods:
    .addEvidence()    append swipe evidence and emit updates
    .addFacade()      add to reservoir with freshness metadata
    .consumeFacade()  remove ready card from the hot path
    .clearReadyFacades() stale queued cards on reveal
    .toEvidencePrompt() serialize evidence for prompts
    .emit() / .on()   session-local event bus
```

The hot path is deliberately small: route validation, evidence append, facade consumption, queue-stat emission, and async scheduling. LLM calls run outside the request path.

## Agent Loops

### Scout Loop (Ralph-style)

```
refill:
  brief <- session.nextProbe OR self-assign from synthesis + queue gaps
  facade <- generate(brief, session snapshot)  // generateText to SCOUT_MODEL
  session.addFacade(facade, reason)            // reservoir owns draw order
  return
```

Scouts are producers. They do not wait for the swipe on the card they generated; swipe handling is session-owned so the user never waits for a scout between cards. Scout identity still matters through lens, assignment, and `agentId`, but readiness belongs to the reservoir.

> Depth: `research/negative-selection.md`, `research/iec-fatigue.md` -- Ralph-style = clonal selection loop. Clone accepted artifacts, mutate at varying step sizes, preserve high-affinity lineages. The dual model (attraction + repulsion) updates on every decision: accepts refine the attraction model; rejects update the repulsion boundary and constrain generation to skip dead regions.

### Builder Loop

```
on every swipe-result:
  enqueue async draft patch
  if accept: integrate surviving artifact into draft
  if reject: add anti-pattern constraint to draft

  ambiguity <- identify what's blocking construction
  if ambiguity exists:
    session.probes.push(ambiguity)      // high-priority brief for scouts

starting around swipe 8:
  prepare shadow reveal material

around swipe 36:
  force reveal-prep pass
```

The builder never generates facades. It reads results and builds asynchronously. Its probe briefs are construction-grounded: not "color axis unresolved" but "I'm building the header and need to know: fixed position or scroll-away, given the resolved dark atmospheric layout with organic texture."

> Depth: `research/active-preference-learning.md` -- Construction-grounded uncertainty > abstract uncertainty. The builder's ambiguities are the system's best proxy for predictive information gain -- what knowledge would actually change the output artifact.

### Oracle Watch

```
on swipe:
  if stage floor changed: emit stage-changed
  if swipeCount % 4 == 0: synthesize evidence
  if synthesis changes: bump tasteVersion
  prune queued facades more than 1-2 taste versions behind
  schedule reservoir refill
  if swipeCount >= 42: enter reveal
```

> Depth: `research/active-preference-learning.md` -- In V0 the LLM handles implicit taste navigation from evidence; coded BALD/fracting remains a future upgrade.

## Queue Equilibrium

Swipe cadence target: as fast as the user can flick. Generation cadence varies by model and stage, so the reservoir must always have facades waiting.

**Buffer target:** warm to at least 12 ready cards before entering swiping. Maintain 16-24 ready cards. Below 8 -> aggressive refill.

**Freshness:** on synthesis update, bump `tasteVersion`. Keep cards within 1-2 versions of current taste when possible; demote or drop older cards unless the deck is near empty.

**Stage blending:** No abrupt transitions. Late word-stage facades can carry small images. Early mockup-stage can be partial layouts. Transitions driven by Anima resolution density.

> Depth: `research/generative-ui.md` -- Design gallery dispersion. Showing N candidates is not UI polish; it is the query design problem. Dispersion and arrangement were formalized as first-class algorithmic subproblems by Marks et al. (SIGGRAPH 1997). Redundancy kills information gain -- if the queue shows perceptually near-identical candidates, every response after the first carries almost no new information.

## Builder-Driven Probes

The builder has the best view of where ambiguity matters. Abstract uncertainty in the Anima tree is less valuable than construction-grounded uncertainty in the builder's draft.

Builder briefs carry resolved constraints as context, so the scout generates facades the user can judge in context, not in the abstract. When the builder is stuck, all scout energy goes to unblocking it. When the builder has no ambiguities, scouts explore freely.

> Depth: `research/active-preference-learning.md` -- Probe = acquisition function query. The builder's briefs are the practical equivalent of construction-grounded information gain. Query-by-committee (QBC) maps to multi-agent setups where models propose candidates and disagreement selects what to show.

## Agent Spawning

Spawning is conditional-VOI gating. Open children only when the parent's entropy is below the lock threshold:

    open children of n only if H_t(n) < h_lock

Then go deeper iff the best child probe score exceeds the best surface probe score plus a margin:

    max child score > max surface score + delta

**Progressive widening** (from MCTS): do not expand a node's action set until it has enough visits to justify expansion. HOO/UCT assign upper-confidence bounds per node and descend toward high-payoff regions. The tree refines where it matters.

**Depth budget `[convention]`:** 2 levels reliable, 3 max on one strong branch, never >3 in a 30-swipe session. Forcing 4-5 levels trains on patience, not taste.

**Retirement:** Queue buffer full + information gain dropping -> retire, don't spawn.

The `spawn-requested` event is emitted by scouts when a parent axis locks and candidate children exist. The oracle gates it: checks parent lock, depth budget, progressive widening threshold, and session swipe budget before honoring the request.

```
Oracle
  +-- Scout A (mood) -> resolved "dark atmospheric"
  |     +-- Scout A1 (texture) -> resolved "organic"
  |     |     +-- Scout A1a (rough vs smooth) -- active
  |     |     +-- Scout A1b (static vs animated) -- active
  |     +-- Scout A2 (density) -- active
  |     +-- Scout A3 (temperature) -> resolved "cool"
  +-- Scout B (layout) -- working builder brief
  +-- Scout C (interaction) -- waiting for mockup stage
  +-- Builder -- 3 sections complete, 2 ambiguities pending
```

**Research basis:** Conditional-VOI from tree-structured BO (Jenatton et al.). CoFineUCB / Hier-UCB for coarse-to-fine explore-exploit. Progressive widening from MCTS literature. Practical depth limits from IEC fatigue bounds (Picbreeder: single-user ~20 generations; cumulative lineages ~151; 98% of top-rated images required >20 cumulative generations).

> Depth: `research/fracting.md`, `research/active-preference-learning.md`

## Anima Compaction

Every 5 swipes or on contradiction:

- **Merge** convergent evidence into single resolved nodes. Must respect branch isolation -- observations from one conditional branch must not leak into another. Use branch-aware factored beliefs: p(Theta|D) = p(branch v|D) * p(Theta_v|D, c_v).

- **Prune** dead branches from strongly resolved axes.

- **Promote contradiction**: axis was misidentified -> re-probe at parent level with reframed axis. This creates a new probe brief with the narrower hypotheses, not just "new probe" -- the compactor must output the reframed axis candidates and their grounding in the contradictory evidence.

- **Summarize** to stay within token budget.

**Between-compaction updates are code** (distribution shifts after each swipe), not LLM. After each swipe, code updates the `exploring` distribution for the targeted dimension: increase probability of the winning hypothesis, decrease the loser, proportional to observation confidence. Compaction (every 5 swipes) is the only LLM rewrite of the full Anima YAML.

> Depth: `research/fracting.md`

## Observation Model

Binary choice + RT -> probabilistic observation via DDM-inspired mapping.

**Research basis:** The Drift-Diffusion Model (DDM) jointly explains RT and choice through three latent parameters: drift rate (evidence accumulation strength), boundary separation (how much evidence before committing), and non-decision time (motor/encoding overhead). Evidence accumulates noisily until hitting one of two boundaries (accept or reject). Which boundary = choice; hitting time + t0 = reaction time.

**Critical insight: RT reflects boundary proximity / decision difficulty, NOT preference strength.** Slow does not mean dislike. Slow means the stimulus sits near the indifference boundary -- evidence was weak or conflicting. Fast accept and fast reject are both high-confidence; slow accept and slow reject are both low-confidence.

**Four-signal interpretation** -- every swipe yields (choice, RT):

| Choice | RT   | Signal            | Action                         |
|--------|------|-------------------|--------------------------------|
| Accept | Fast | Strong positive   | Amplify region                 |
| Accept | Slow | Weak / curious    | Near boundary; probe neighbors |
| Reject | Fast | Strong negative   | Suppress region                |
| Reject | Slow | Weak / unsure     | Near boundary; probe neighbors |

The slow cases are the most valuable for learning because they identify the indifference surface.

**JND floor as stopping criterion:** The just-noticeable difference is the smallest change a user can reliably detect. Once remaining candidate differences fall below the user's stable JND, further optimization produces imperceptible changes -- stop refining, lock the result, inject novelty, or shift axes. The JND floor is local, not universal: it varies by axis, by region (sharpens near category boundaries), by user expertise, and by session fatigue state. Weber's Law: step sizes should scale proportionally with current position.

**Mere exposure warning:** Repeated exposure increases liking. The system shapes preference while measuring it. Without controls, the posterior drifts toward "familiar" not "preferred." Mitigate: novelty injection, time-decay on old evidence, inverse-exposure weighting on posterior updates, explicit familiarity modeling. Distinguish "preferred because closer to target" from "preferred because seen three times."

**`[convention]`** The specific RT -> confidence mapping function is a hackathon convention. RT calibration: estimate t0 from the floor of the session's RT distribution (fastest observed responses); decision time = RT - t0. Full DDM parameter fitting deferred. See observation-model topic file for formal likelihood and psychometric function details.

> Depth: `research/observation-model.md`

## The Living Prototype

The builder doesn't assemble at the end. It grows the prototype behind every swipe.

**Three inputs:** Surviving artifacts (material), the Anima tree (rules), rejected artifacts (guardrails).

**Gap filling:** Unresolved sections filled by extrapolation from Anima. The taste profile is a gravitational field. The builder fills the dark matter.

**Reveal:** User finishes swiping. Prototype is already there. Grown, not generated.

> Depth: `research/observation-model.md` -- Mere exposure warning. The prototype itself becomes a source of familiarity bias. Sections built from early swipes have more exposure; late sections are fresher. Weight accordingly during final assembly.
